One of the most interesting and widely discussed trends in early 2026 is not about a new model – it is about **pushback**.

Several long-running open source projects and communities have started to **explicitly ban AI-generated code contributions**, citing concerns about quality, licensing, and long-term maintainability.

Today’s brief looks at what is happening, why maintainers are drawing a line, and what this means if you are using AI tools to write code.

---

### 1. What Is Actually Happening?

According to recent reports and maintainer statements:

- Projects like **Gentoo Linux** and **NetBSD**, among others, have published guidelines or policies that:
  - Discourage or outright **forbid AI-generated patches** from being submitted
  - Ask contributors to clearly disclose when AI tools were used
  - Emphasise the need for human understanding and accountability

This is not a blanket rejection of AI tools, but it is a clear signal:

> "We want contributors, not just output."  
> "We want maintainable code, not paste-ins." 

---

### 2. Why Are Maintainers Pushing Back?

Several reasons keep coming up in discussions and policy notes:

**a) Quality and correctness**  
AI-generated code can look polished but still:

- Misunderstand edge cases
- Introduce subtle bugs
- Duplicate logic that already exists elsewhere in the codebase

For maintainers who need to support a project over many years, "mysterious" code is a liability.

**b) Licensing and provenance**  
If a model was trained on a mix of code with different licenses, it can be difficult to:

- Prove that an AI-generated snippet is safe to include
- Trace where patterns or fragments came from

Open source projects are often very careful about license compatibility – this uncertainty is unacceptable for some.

**c) Learning and community**  
Open source has always been a place where newcomers learn by:

- Reading existing code
- Tackling small bugs
- Getting feedback from maintainers

If contributions become mostly "AI wrote this", it weakens the learning loop and the sense of craft.

---

### 3. What This Means If You Use AI to Code

If you use tools like GitHub Copilot, ChatGPT, or other AI assistants when coding, this does **not** mean you must stop.

But it does mean:

- You are still responsible for understanding the code you submit
- You should be prepared to explain:
  - Why this approach is correct
  - How it fits into the rest of the project
  - What alternatives you considered

Some practical guidelines for contributing to open source in this new environment:

1. **Use AI as a helper, not a patch generator**  
   - Let it suggest ideas or boilerplate, but reshape and refactor yourself

2. **Always read and rework the output**  
   - Do not submit anything you would not be able to maintain on your own

3. **Be transparent when asked**  
   - If a project has policies about AI assistance, follow them and disclose when appropriate

---

### 4. The Bigger Picture: Hybrid Craft

The open source pushback is not a rejection of AI – it is a reminder of something deeper:

- High-quality software is still a **human craft**
- Tools can speed up parts of the process, but ownership and responsibility remain with people

For individual developers and small teams, the sweet spot is likely to be:

> AI for speed – human judgement for structure and long-term health.
